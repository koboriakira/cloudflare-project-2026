## 設計判断（サンプル）

以下はテンプレート採用時点の判断をサンプルとして残したもの。実プロジェクトの判断に置き換えてよい。

### なぜ Hono + zod-openapi か

- Cloudflare Workers の V8 isolate ランタイムで軽量に動くルーターが必要だった
- @hono/zod-openapi により、ルート定義に Zod スキーマを埋め込むことで API 契約と実装の乖離を構造的に不可能にする（code-first 戦略）
- openapi.yaml はコードから生成する派生物であり、正本ではない
- トレードオフ: 実装前に API 契約を合意する design-first フローが使えない。個人プロジェクトのため問題なし

### なぜ D1 か

- Cloudflare が提供するマネージド SQLite 互換 DB で、Worker と同一プラットフォーム内でレイテンシが低い
- トレードオフ: 外部 DB に比べスケール上限に制約があるが、初期構成のシンプルさを優先

### なぜ pool-workers か

- テスト戦略「統合テスト主軸」を実現するため
- Miniflare 上で実際のバインディング（D1, R2）を使ったテストが書ける
- モック unit テストは結合部の不整合を検出できないため低価値と判断

## アーキテクチャ

```
src/
  app.ts         — OpenAPIHono アプリ定義（ルート結合）
  index.ts       — Workers エントリポイント（Sentry ラップ）
  schemas/       — Zod スキーマ（API 契約の正本）
  routes/        — ルートハンドラ（スキーマ参照）
  services/      — ビジネスロジック
  types/         — 型定義（Bindings 等）
  middleware/    — ミドルウェア
scripts/
  generate-openapi.ts — openapi.yaml 生成
migrations/      — D1 マイグレーション
judges/          — 不変条件の機械検証
```
