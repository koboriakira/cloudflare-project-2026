# API 契約の同期整合性

## 守るべき不変条件

`openapi.yaml`（API 契約の正本）と `src/routes/` 以下の実装について、以下が常に真であること。

1. **`openapi.yaml` の `paths` に定義された各パスに対応するハンドラが `src/routes/` に存在する** — 契約にあって実装にないエンドポイントを放置しない
2. **`src/routes/` に新しいエンドポイントを追加したら `openapi.yaml` も同時に更新する** — 実装が正本を追い越した状態を作らない
3. **正本を更新せずに実装だけ変更した場合、次のコミット前に `openapi.yaml` を追いつかせる**

## なぜこのルールが必要か

- `openapi.yaml` は型生成・CI・クライアント向けドキュメントの入力元になる。実装との乖離は、生成された型が実際の API 挙動と一致しない事故に直結する
- API 契約は「決めてから実装する」場合と「実装しながら固める」場合の両方があるが、いずれの順序でも最終的に両者が一致していることを機械的に確認できないと、乖離は静かに蓄積する
- エンドポイント数のような軽量な指標であっても、乖離の兆候を早期に検出できれば、詳細レビューのコストを大きく下げられる

## 違反時の対応

`judges/openapi-sync.sh` で検出する。完全なスキーマ検証（リクエスト/レスポンス形状の一致）は行わず、パス数の一致など軽量なチェックに留める。詳細な整合性はコードレビューで確認する。
